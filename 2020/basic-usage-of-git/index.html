使用Git进行版本控制

入门 2020-01-06 784 字 2 分钟

　　入职了一家小公司，后端就仨菜鸡还裁了一个，比上不足比下有余的老陈就担负起了管理代码的重担。虽然说一直都在使用 git，但之前也仅仅局限在本地往测试服务器提交代码，更新到正式服务器都是由主管/组长负责，经常接触的命令也只有

  1. git status
  2. git commit
  3. git fetch
  4. git pull
  5. git push

　　突然从一对一变成一对多让我手足无措了一阵子，百度上找了好几篇文章看的云里雾里。上头催得紧实在是没有时间去细品更希望有一篇简洁明了的文章告诉我怎么去做。所以在此做一个记录，希望某一天可以帮助到另外一个迷茫的小菜鸡。

　　代码一般托管在 github 或者码云上，即远程仓库 remote，仓库里有两个分支，主分支master 和开发分支 dev。你的手上有两台服务器，一台测试，一台正式。

首先，你在本地进行开发，确认你在 dev 分支上：

git branch

如果不在 dev 分支则切换到 dev 分支：

git checkout dev

接下来愉快地写 bug，本地测试完成后进行提交：

git status git add . git commit -m 'update msg'

git fetch git pull git push

此时你的代码以及更新到远程仓库 remote 的 dev 分支中

然后在测试服务器上拉取 remote 的 dev 分支代码：

使用 xshell 连接，确认当前在 dev 分支，执行git pull

代码就更新到了测试服务器，进行测试。测试通过后进行合并，本地执行：

git checkout master ：切换到 master 分支

git merge dev ：将 dev 上的改动合并到 master 分支

git pull ：将合并后的代码推送到 remote 的 master 分支

然后在正式服务器上拉取 remote 的 master 分支代码：

使用 xshell 连接，确认当前在 master 分支，执行git pull

期间如果出现问题，按照 git 给出的提示进行操作。

要点：

  • 如非必要，测试及生产只执行 git pull 命令
  • 干活前，先确认当前使用的分支 git branch
  • 如果遇到本地通过，测试/正式报错，某某文件权限问题，不要直接 chmod 777 filename 因为 git 会把文件权限也列入了版本管理，下次更新代码执行 git pull 会报错，让你先提交。正确姿势是先用命令 git config core.filemode false 取消 git 对文件权限的管理再修改文件权限

　　简单的一点点问题这两天把老吴折腾得够呛。如果在杭州的正常规模的公司，问一下旁边的老哥，五分钟解决，所以来义乌真是近两年最错误的决定 : (
